Context

The Vehicle Routing Problem (VRP)

Every morning, a dispatcher must answer:
"Which driver visits which customers, in what order?"

The naive approach

  • Assign orders sequentially as they arrive
  • No geographic clustering
  • Ignore time windows & capacity
  • Result: trucks criss-cross the city

The optimised approach

  • Cluster stops spatially by depot
  • Nearest-neighbour with time-window feasibility
  • Respect payload capacity
  • Result: tight geographic loops

-- Step 1: Register spatial metadata
--   (defines bounding box for Chicago metro)
INSERT INTO user_sdo_geom_metadata VALUES (
  'FLEET_CUSTOMERS',    -- table
  'LOCATION',           -- column
  SDO_DIM_ARRAY(
    SDO_DIM_ELEMENT('LON',-88.50,-87.30,0.005),
    SDO_DIM_ELEMENT('LAT', 41.60, 42.20,0.005)
  ),
  4326                  -- SRID
);

-- Step 2: Create spatial index
CREATE INDEX idx_fleet_customers_loc
  ON fleet_customers(location)
  INDEXTYPE IS MDSYS.SPATIAL_INDEX_V2
  PARAMETERS ('layer_gtype=POINT');

-- Step 3: Also create VECTOR index for AI
CREATE VECTOR INDEX idx_fleet_ro_vector
  ON fleet_routes_optimized(route_vector)
  ORGANIZATION INMEMORY NEIGHBOR GRAPH
  DISTANCE COSINE
  WITH TARGET ACCURACY 95;
-- ↑ Oracle 26ai HNSW approximate nearest-neighbour
  • R-Tree index: each node stores a Minimum Bounding Rectangle (MBR) covering a subset of geometries. A proximity query does a two-pass search — filter (MBR overlap) then refine (exact geometry test). Only candidates whose MBR overlaps the query region are refined.
  • USER_SDO_GEOM_METADATA must be populated before CREATE INDEX. It registers the bounding box (Chicago metro) and SRID. Forgetting this step is the #1 SDO setup error.
  • Eight spatial indexes in this demo — one per spatial column. Each enables different operator pairs in the analysis queries: SDO_NN on depots, SDO_WITHIN_DISTANCE on customers, SDO_INSIDE on traffic zones.
  • SPATIAL_INDEX_V2 is the current generation index (Oracle 12.2+). V2 supports geodetic coordinates (SRID 4326) natively, handles antimeridian crossing, and supports partitioned tables.
  • The HNSW VECTOR index on the same slide shares session memory with the R-Tree — both are native Oracle index types, managed under the same transaction and backup model.

Oracle Spatial · Queries

SDO_NN & SDO_WITHIN_DISTANCE

-- ── Q1: Nearest depot to each customer ────────
-- SDO_NN = Spatial Nearest Neighbour
-- Uses R-Tree index automatically
SELECT c.customer_name,
       d.depot_name,
       ROUND(SDO_GEOM.SDO_DISTANCE(
           c.location, d.location,
           0.005, 'unit=KM'
       ), 2)  AS dist_km
FROM   fleet_customers c,
       fleet_depots    d
WHERE  SDO_NN(
           d.location,      -- geometry to search
           c.location,      -- reference point
           'sdo_num_res=1', -- return 1 nearest
           1
       ) = 'TRUE'
ORDER  BY c.customer_id;

-- ── Q2: All customers within 5 km of O'Hare ──
SELECT c.customer_name, c.neighborhood
FROM   fleet_customers c
WHERE  SDO_WITHIN_DISTANCE(
           c.location,
           (SELECT location FROM fleet_depots
            WHERE depot_name LIKE '%O''Hare%'),
           'distance=5 unit=KM'
       ) = 'TRUE';

-- ── Q3: Vehicles inside CBD traffic zone ──────
SELECT v.vehicle_code, t.speed_kmh
FROM   fleet_vehicles v
JOIN   vw_vehicle_locations t USING(vehicle_id)
JOIN   fleet_traffic_zones  tz ON tz.zone_type='CBD'
WHERE  SDO_INSIDE(
           t.current_location,
           tz.zone_boundary        -- polygon
       ) = 'TRUE';

SDO_NN — Nearest Neighbour

Returns the N closest geometries. Used in the VRP engine to find the nearest unvisited customer at each step.

SDO_WITHIN_DISTANCE

Returns all geometries within a radius. Used to check if a delivery stop falls inside a congestion zone.

SDO_INSIDE / SDO_GEOM

Point-in-polygon (traffic zone containment). SDO_GEOM.SDO_DISTANCE returns exact km between any two geometries.

  • SDO_WITHIN_DISTANCE uses a two-pass R-Tree approach: pass 1 (filter) finds MBR candidates; pass 2 (refine) applies exact distance. The tolerance parameter (0.005 degrees ≈ 500 m) controls filter precision — never produces false negatives.
  • SDO_NN traverses the R-Tree outward from the reference point until sdo_num_res neighbours are found. The second numeric argument (1) is the batch hint — Oracle uses it to optimise the query plan for joins. Setting sdo_num_res=1 returns the single nearest neighbour.
  • SDO_INSIDE is a point-in-polygon test. Oracle uses a ray-casting algorithm after the R-Tree filter narrows candidates to polygons whose MBR contains the point.
  • All three operators return the string 'TRUE' for matching rows — this is Oracle Spatial convention. The = 'TRUE' predicate must be present for the spatial index to be used.
  • These three operators map to the three VRP use cases: depot assignment (SDO_NN), zone surcharge detection (SDO_WITHIN_DISTANCE), and CBD congestion check (SDO_INSIDE).

Oracle 26ai Feature 2

AI Vector SearchVECTOR type

Oracle 26ai introduces a native VECTOR data type — enabling AI similarity search directly inside the relational database. No separate vector database needed.

What we store

Each optimised route gets a 128-dimensional floating-point embedding — a mathematical fingerprint encoding:

  • Route centroid (geographic centre)
  • Geographic spread of stops
  • Total distance & stop count
  • Depot assignment
  • Harmonic spatial features

What it enables

"Find me routes similar to this one from last month"
— answered in milliseconds via cosine similarity on HNSW index

-- ── VECTOR column definition ──────────────────
CREATE TABLE fleet_routes_optimized (
  route_id   NUMBER PRIMARY KEY,
  vehicle_id NUMBER,
  ...
  -- Oracle 26ai: 128-dim float32 vector
  route_vector  VECTOR(128, FLOAT32)
);

-- ── Build HNSW vector index ───────────────────
-- HNSW = Hierarchical Navigable Small World
-- Approximate nearest-neighbour, very fast
CREATE VECTOR INDEX idx_fleet_ro_vector
  ON fleet_routes_optimized(route_vector)
  ORGANIZATION INMEMORY NEIGHBOR GRAPH
  DISTANCE COSINE
  WITH TARGET ACCURACY 95;

-- ── Store a vector (from PL/SQL) ──────────────
UPDATE fleet_routes_optimized
SET    route_vector = TO_VECTOR(
         sys.ODCINumberList(
           -0.071,  0.032,  0.184,  -- dims 1-3
           ...                      -- dims 4-128
         ),
         128, FLOAT32
       )
WHERE  route_id = :id;

-- ── Query: find 5 most similar routes ────────
SELECT route_id, VECTOR_DISTANCE(
         route_vector, :query_vec, COSINE
       ) AS similarity
FROM   fleet_routes_optimized
ORDER  BY similarity
FETCH  FIRST 5 ROWS ONLY;
  • HNSW (Hierarchical Navigable Small World) pre-builds a multi-layer graph. The top layer is a sparse "highway" graph; lower layers get denser. Query enters the top layer and greedily descends, collecting nearest-neighbour candidates — O(log n) vs O(n) brute force.
  • VECTOR(128, FLOAT32) = 512 bytes per route, stored natively in the Oracle row. No separate vector store, no ETL, no synchronisation. The vector and the relational columns are in the same data block.
  • Cosine similarity = 1 − cosine_distance. Score 1.0 = identical route (same route compared to itself). Score ~0.95 = same depot, similar neighbourhood cluster. Score < 0.80 = geometrically dissimilar routes.
  • 128 dimensions encode: centroid lat/lon, geographic spread, stop count, total distance, depot (one-hot), time-window stats, directional bearing features, Fourier-style shape harmonics.
  • TARGET ACCURACY 95 means the HNSW search returns 95% of the true nearest neighbours on average. For fleet routing use cases, 5% recall loss is operationally acceptable and delivers 10-100x speed improvement over exact search.

-- ── Oracle 26ai: VECTOR similarity search ────
-- Find 5 routes most similar to route #1
-- HNSW index makes this sub-millisecond

SELECT
    route_id,
    vehicle_id,
    num_stops,
    ROUND(total_distance_km, 1)   AS dist_km,
    ROUND(savings_pct, 1)         AS saved_pct,
    -- Similarity score: 1.0 = identical, 0 = unrelated
    ROUND(
      1 - VECTOR_DISTANCE(
            route_vector,
            (SELECT route_vector
             FROM   fleet_routes_optimized
             WHERE  route_id = 1),  -- reference route
            COSINE
          ),
      3)  AS similarity_score
FROM   fleet_routes_optimized
WHERE  route_vector IS NOT NULL
ORDER  BY similarity_score DESC
FETCH  FIRST 5 ROWS ONLY;

-- Example output:
-- ROUTE_ID | DIST_KM | SAVED_PCT | SIMILARITY
-- ─────────────────────────────────────────────
--        1 |    47.3 |      34.2 |      1.000  ← self
--        4 |    43.8 |      31.7 |      0.982
--        7 |    51.2 |      29.4 |      0.967
--        2 |    39.1 |      35.8 |      0.941
--       11 |    44.7 |      33.1 |      0.928
  • The ORDER BY VECTOR_DISTANCE(...) pattern is what triggers the HNSW index. Oracle recognises this exact form and uses the graph for approximate nearest-neighbour traversal rather than full scan.
  • Score 1.0 is the reference route compared to itself — always the top result. Scores 0.928–0.982 correspond to routes from the same depot serving geographically adjacent zones.
  • Use cases: find historical routes matching today's orders (reuse past optimisations), identify drivers familiar with a route pattern, detect anomalous routes that score < 0.70 against all historical routes.
  • HNSW vs exact: exact brute-force is O(n) per query; HNSW is O(log n). At 1,000 stored routes the difference is small; at 1 million historical routes (2 years of a large fleet) HNSW is 1000x faster.

Oracle 26ai Feature 3b

JSON Relational Duality ViewsOracle 26ai

A Duality View exposes relational tables as JSON documents — giving apps a document API while the database stores and enforces relational constraints underneath. No duplication. No sync lag.

orders_duality

Joins fleet_orders, fleet_customers, and fleet_depots into a single order document with full INSERT / UPDATE / DELETE support.

Use cases

  • Mobile dispatch app reads orders as JSON
  • REST API via ORDS / Database Actions
  • Claude MCP agent queries as documents
  • Relational consistency maintained automatically
-- ── JSON Relational Duality View ─────────────
CREATE OR REPLACE JSON RELATIONAL DUALITY VIEW
  orders_duality AS
SELECT JSON {
  '_id'          : o.order_id,
  'orderRef'     : o.order_ref,
  'deliveryDate' : o.delivery_date,
  'status'       : o.status,
  'priority'     : o.priority,
  'payload'      : {
      'weightKg' : o.weight_kg,
      'volumeM3' : o.volume_m3
  },
  'customer'     : {
      'id'           : c.customer_id,
      'name'         : c.customer_name,
      'neighborhood' : c.neighborhood
  },
  'depot'        : {
      'id'   : d.depot_id,
      'name' : d.depot_name
  }
}
FROM  fleet_orders    o
JOIN  fleet_customers c ON c.customer_id = o.customer_id
JOIN  fleet_depots    d ON d.depot_id    = o.depot_id
WITH INSERT UPDATE DELETE;  -- ← full read-write!

-- ── Query like a document store ───────────────
SELECT od.data
FROM   orders_duality od
WHERE  JSON_VALUE(od.data,'$.status')   = 'ASSIGNED'
AND    JSON_VALUE(od.data,'$.priority') = '1';
  • No JSON is stored. The duality view assembles the document at query time from B-Tree lookups across fleet_orders, fleet_customers, and fleet_depots. If a customer address changes, the next query returns the updated document instantly — no ETL or cache refresh needed.
  • Write-through decomposition: an INSERT or UPDATE through the duality view is automatically decomposed into relational DML. Oracle checks all FK constraints and NOT NULL rules against the underlying tables. The WITH INSERT UPDATE DELETE clause makes the view fully bidirectional.
  • This is not the same as SODA collections: SODA stores actual JSON documents. Duality views project relational data as JSON — the relational schema is always the source of truth.
  • MongoDB Compass can connect to Oracle 26ai's MongoDB-compatible endpoint and browse orders_duality as a collection. A find({status:"ASSIGNED"}) is transparently translated to the SQL query against the duality view.
  • Key talking point: "Same physical data — three different apps see it as rows, as a REST resource via ORDS, and as a MongoDB document. Zero synchronisation lag between them."

Fleet telemetry is a high-volume time-series workload — potentially millions of GPS pings per day. Range partitioning on recorded_at lets Oracle prune old partitions from queries automatically.

Query performance impact

A query for "today's pings" scans 1 partition instead of the full table.
With 2 years of history, that's a 8× speedup.

Partitions defined

  • telem_2025_q1  — Jan–Mar 2025
  • telem_2025_q2  — Apr–Jun 2025
  • telem_2025_q3  — Jul–Sep 2025

Oracle 26ai Feature 5

PL/SQL — VRP AlgorithmServer-side logic

The entire VRP solver runs inside the database as PL/SQL — no data movement, no middleware. Spatial distance calculations use the Haversine formula on longitude/latitude pairs.

Algorithm: Nearest-Neighbour Heuristic

  1. Sort orders by priority then time-window open
  2. Assign orders to their nearest depot spatially
  3. For each vehicle: greedy loop — pick nearest feasible unvisited stop
  4. Feasibility = within time-window AND within payload capacity
  5. Priority boost: score = distance / (4 − priority)
  6. Return to depot, compute fuel cost & CO₂
  7. Store 128-dim VECTOR embedding for AI search
-- ── Haversine distance (km) in PL/SQL ────────
FUNCTION haversine(
    lon1 NUMBER, lat1 NUMBER,
    lon2 NUMBER, lat2 NUMBER
) RETURN NUMBER IS
  r  CONSTANT NUMBER := 6371; -- Earth radius km
  dl  NUMBER := (lat2-lat1) * 3.14159/180;
  dlo NUMBER := (lon2-lon1) * 3.14159/180;
  a   NUMBER;
BEGIN
  a := SIN(dl/2)*SIN(dl/2)
     + COS(lat1*3.14159/180)
     * COS(lat2*3.14159/180)
     * SIN(dlo/2)*SIN(dlo/2);
  RETURN r * 2 * ATAN2(SQRT(a), SQRT(1-a));
END;

-- ── Nearest-feasible-neighbour loop ──────────
LOOP
  v_best_idx  := 0;
  v_best_dist := 999999;

  FOR oi IN 1..v_ord_cnt LOOP
    IF NOT v_ord_served(oi)
    AND (v_payload + v_weight(oi)) <= v_max_pay
    AND arrival_minutes <= v_time_window_close(oi)
    THEN
      -- Priority-weighted score
      v_score := haversine(cur_lon, cur_lat,
                           ord_lon(oi), ord_lat(oi))
                 / (4 - v_priority(oi));
      IF v_score < v_best_dist THEN
        v_best_dist := v_score;
        v_best_idx  := oi;
      END IF;
    END IF;
  END LOOP;
  EXIT WHEN v_best_idx = 0;
  -- commit to this stop, mark served ...
END LOOP;

Baseline Routes — The Problem State

Script 03_baseline_routes.sql simulates how deliveries look before any optimisation — orders assigned sequentially by order_id, not geography.

  • Distribute round-robin across 15 vehicles
  • No geographic clustering
  • No time-window checking

  • Routes criss-cross the city
  • Vehicles from NW depot serve South Side customers

-- ── Naive order assignment ────────────────────
-- Orders sorted by order_id (arrival order)
-- NO geographic awareness
FOR r IN (
  SELECT o.order_id,
         c.location.sdo_point.x  clon,
         c.location.sdo_point.y  clat
  FROM   fleet_orders   o
  JOIN   fleet_customers c ON ...
  WHERE  o.delivery_date = c_demo_date
  ORDER  BY o.order_id   -- ← NAIVE: no clustering
) LOOP
  -- Assign: vehicle 1 gets orders 1,16,31,46...
  --         vehicle 2 gets orders 2,17,32,47...
  --         etc. (round-robin)
  IF MOD(o_idx-1, v_veh_cnt)+1 = v_idx THEN
    -- compute haversine distance from prev stop
    v_dist := haversine(prev_lon, prev_lat,
                        cust_lon,  cust_lat)
              * 1.35;  -- urban factor
    -- insert stop record...
  END IF;
END LOOP;

-- ── Typical baseline route output ────────────
-- Route 1 (OHR-HT-01): 5 stops, 71.4 km, $27.13
--   Stop 1: O'Hare → Woodlawn (38 km south)
--   Stop 2: Woodlawn → Park Ridge (62 km north)
--   Stop 3: Park Ridge → Hyde Park (55 km south)
--   ← criss-crossing the entire city!
  • Haversine formula: a = sin²(Δlat/2) + cos(lat1)·cos(lat2)·sin²(Δlon/2); d = 6371 × 2·atan2(√a, √(1−a)). Great-circle distance — same math inside a GPS receiver.
  • The ×1.35 urban factor converts crow-fly km to road-km. Chicago's street grid means actual driving is 25–35% longer than straight-line distance.
  • Round-robin: vehicle 1 gets orders 1, 16, 31, 46… — consecutive order IDs span the entire metro. Each truck zig-zags city-wide because arrival time has no geographic correlation.
  • All 15 vehicles dispatched, ~855 km total, ~$2,420/day. Three vehicles are wasted due purely to the assignment method.
  • Key line: "The geographic chaos is invisible on a spreadsheet but shows up immediately in a fuel bill."

Demo Step 4

VRP Optimisation — The Solution

Script 04_vrp_optimize.sql — same data, completely different outcome. The PL/SQL solver clusters stops geographically, respects time windows, and only dispatches vehicles that are needed.

Key improvements

  • Orders sorted: priority first, then time-window open
  • Each order assigned to nearest depot spatially
  • Nearest-neighbour with priority weighting
  • Time-window feasibility checked per step
  • Payload cap per vehicle enforced
  • VECTOR embedding stored for each completed route

Result

  • Routes form tight geographic loops
  • 3 vehicles not needed (freed for other tasks)
  • ~575 km total · ~$1,625 fuel
-- ── Key difference: orders pre-sorted ────────
FOR r IN (
  SELECT o.order_id, ...
  FROM   fleet_orders o
  JOIN   fleet_customers c ON ...
  WHERE  o.delivery_date = c_demo_date
  ORDER  BY o.priority,          -- ← URGENT first
            c.time_window_open,  -- ← early windows first
            o.order_id
) LOOP ...

-- ── Nearest-feasible-neighbour selection ──────
FOR oi IN 1..v_ord_cnt LOOP
  IF NOT served(oi)
  AND nearest_depot(oi) = this_depot
  AND payload + weight(oi) <= max_payload
  AND arrival_minutes    <= time_window_close(oi)
  THEN
    -- Priority-weighted score
    v_score := haversine(cur_pos, stop_pos)
               / (4 - priority(oi));
    -- Lower score = pick this one next
  END IF;
END LOOP;

-- ── After building route: store VECTOR ───────
UPDATE fleet_routes_optimized
SET    route_vector = make_route_vector(
           centroid_lon, centroid_lat,
           geographic_spread,
           total_distance,
           num_stops,
           depot_id
       )
WHERE  route_id = v_route_id;
  • Nearest-neighbour heuristic: at each step, scan all unserved feasible orders and pick the lowest score. O(n²) per vehicle — fast enough for 75 orders, within 20-25% of optimal.
  • Priority score formula: score = haversine(current, stop) / (4 − priority). Priority 1 URGENT gets a ÷3 discount — it appears 3× closer than it is, so it is picked first.
  • Two feasibility gates per candidate: (1) payload + order_weight ≤ vehicle_max_payload, (2) estimated_arrival ≤ time_window_close. Both must pass or the stop is skipped this iteration.
  • VECTOR embedding: 128 floats encode route centroid, geographic spread, stop count, total distance, depot, and Fourier-style directional features. Two routes with similar footprint get cosine similarity > 0.90.
  • Result: same 75 orders, 12 vehicles instead of 15, ~575 km vs ~855 km. Three vehicles are freed — available for other routes or maintenance.

Demo Step 5

Real-time Telemetry Stream

Script 05_telemetry_stream.sql simulates a live GPS data feed — vehicles reporting position, speed, heading, and fuel every 5 minutes as they execute the optimised routes.

What each ping records

  • location — SDO_GEOMETRY point (live GPS)
  • speed_kmh — realistic: 10–65 km/h, 0 at stops
  • heading_deg — compass bearing 0–360°
  • fuel_level_pct — depletes along route
  • engine_temp_c — with random variation
  • status — DRIVING / IDLE / DELIVERING
  • alert_code — FUEL / TEMP / SPD (threshold alerts)

Stored in partitioned table

All pings automatically route to the correct time partition. Spatial index enables instant "which vehicles are near X?" queries.

-- ── GPS ping interpolated along route ────────
-- For each segment (stop A → stop B):
--   emit one ping every 5 minutes

WHILE elapsed_min < segment_min LOOP
  v_t   := elapsed / total_seg_min;  -- 0.0 → 1.0
  v_lon := lerp(lon_from, lon_to, v_t);
  v_lat := lerp(lat_from, lat_to, v_t);

  -- Realistic speed: slower near stops
  IF v_t < 0.1 OR v_t > 0.9 THEN
    v_speed := DBMS_RANDOM.VALUE(10, 25); -- near stop
  ELSE
    v_speed := DBMS_RANDOM.VALUE(35, 65); -- mid-segment
  END IF;

  -- Auto-generate alerts
  v_alert := CASE
    WHEN fuel_level < 15 THEN 'FUEL'
    WHEN engine_temp > 95 THEN 'TEMP'
    WHEN v_speed     > 90 THEN 'SPD'
    ELSE NULL
  END;

  INSERT INTO fleet_telemetry (
    vehicle_id, recorded_at,
    location,                  -- SDO_GEOMETRY point
    speed_kmh, heading_deg,
    fuel_level_pct, engine_temp_c,
    status, alert_code
  ) VALUES (...);

  elapsed := elapsed + 5;   -- next ping in 5 min
END LOOP;
  • Range partition routing is automatic: Oracle inspects the recorded_at timestamp and writes to the correct quarterly partition with no application logic. Query with WHERE recorded_at >= TRUNC(SYSDATE) and Oracle eliminates 5 of 6 partitions before touching a single row.
  • Spatial index on fleet_telemetry.location works per-partition. "Vehicles within 2 km of depot right now" is a spatial scan of only the current-quarter partition — sub-10ms at 2M rows.
  • Status state machine: IDLE (at depot) → DRIVING (in transit) → DELIVERING (at customer, speed=0). Transition logic is pure position-in-route arithmetic.
  • Alert thresholds: FUEL when fuel_level_pct < 15, TEMP when engine_temp_c > 95, SPD when speed_kmh > 90. Generated deterministically — no random noise on alert generation.
  • In production at ~200 pings/vehicle/day this table grows to 2.1M rows over 2 years. Partition pruning keeps queries at the same speed regardless of historical volume.
